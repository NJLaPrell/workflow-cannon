## Purpose

Define **stable artifact IDs**, **artifact types** (v1 + reserved `cognitive-map`), discovery, validation, and how markdown/docs are referenced without inlining content in bundles.

## Scope

**In:** Registry layout (files vs DB), ID format, immutability/aliases, type enum, validator behavior for reserved types.  
**Out:** Exhaustive inventory of every doc in repo (see T857).

## Deliverable artifacts

1. **ADR** (registry & ID conventions).
2. **Normative evolution** of stub schema [`../artifacts/stub-registry-entry.schema.json`](../artifacts/stub-registry-entry.schema.json) → checked-in v1 schema under `schemas/` (path TBD).
3. **Examples**: 3 valid + 3 invalid registry entries (fixtures for later tests).

## Decisions to resolve

- JSON vs YAML vs split-file registry on disk.
- Global vs layered registries (kit default + workspace overlay).
- Whether `cognitive-map` entries are **rejected** in v1 validators or **accepted-but-inert**.

## Acceptance criteria

- [ ] Any implementer can validate `artifactId` and `artifactType` without reading code.
- [ ] Reserved `cognitive-map` behavior is explicit.

## Implementation notes

Keep artifact bodies in git; CAE stores **references** only.
