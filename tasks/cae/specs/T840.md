## Purpose

Specify machine-readable **activation definitions**: families, scopes, priorities, artifact references, flags — with **no** arbitrary code, **no** freeform NL conditions, **no** workflow macros.

## Scope

**In:** Versioned schema, validation rules, minimal examples per family, explicit prohibitions.  
**Out:** Evaluator code, persistence migrations.

## Deliverable artifacts

1. **`schemas/cae/activation-definition.schema.json`** — create the `schemas/cae/` directory if absent; align with existing repo JSON Schema conventions (e.g. `$schema`, `$id` URLs under `workflow-cannon.dev` if used elsewhere).
2. **Fixtures**: `fixtures/cae/activations/*.json` — one minimal valid per family + invalid cases (malformed scope, illegal type, embedded script string if testing rejection).
3. **Mapping table**: activation field → trace event types that must fire when evaluated (feeds T846).

## Decisions to resolve

- JSON Schema draft vs Zod-from-TS as source of truth.
- **Structured condition** enum for v1 (if any): e.g. command name match, task tag match — exact list.
- Schema version field naming (`schemaVersion` vs `$schema` + file version).

## Acceptance criteria

- [ ] CI can validate fixtures against schema (wire in T858 or follow-up).
- [ ] Schema text explicitly forbids executable content fields.
