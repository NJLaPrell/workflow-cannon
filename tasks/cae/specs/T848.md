## Purpose

Define **shadow** vs **live** evaluation labels on the same code path: **would activate / would require / would enforce**; conflict/shadow summary; **usefulness vs noise** capture hooks.

## Scope

**In:** Flags/env names, output fields on bundle or sidecar `shadow` object, persistence optional (see T845).  
**Out:** Enforcement (T866).

## Deliverable artifacts

1. **Design doc** `.ai/cae/shadow-mode.md` with JSON examples.
2. **Schema extension** to bundle or envelope: `mode: "shadow" | "live"`, `shadowSummary: { ... }`.
3. **Matrix**: mode × command × runtime hook behavior.

## Decisions to resolve

- Default **shadow-on** for N releases vs opt-in env.
- Who sets **useful/noisy** (operator CLI, file, absent in v1).
- Whether shadow runs always persist trace (T867).

## Acceptance criteria

- [ ] Shadow cannot weaken code invariants (explicit in doc).
- [ ] T863 implementer can label outputs without new merge logic.

## Implementation notes

Shadow should reuse T860 evaluator with a **mode flag** affecting labels and enforcement hooks only.
